# /triage-bug

Converts a user-reported bug into an engineering-ready Jira ticket. Takes
raw user language (from in-app reports, emails, support tickets, manual
paste-ins) and produces a structured ticket with inferred root cause, file
paths, severity, and test cases.

This is triage, not implementation. Output is reviewed by a human before
creation.

## Usage

```
/triage-bug                          # prompts for the report text
/triage-bug --source ticket:SCRUM-123  # triage from an existing ticket (support → bug)
/triage-bug --source email:thread-id   # from Gmail thread (requires Gmail MCP)
/triage-bug --source paste             # paste-in mode; user pastes raw report text
/triage-bug --apply                    # skip human-review step and create immediately (use with caution)
```

Default is `--source paste` in interactive mode, no `--apply` — produces a
draft and stops for review.

## What you (Claude) must do, step by step

### 1. Collect the raw user report

Based on source flag:

- **paste (default)**: ask user to paste the raw report. Expect
  plain-English user complaint — could be one sentence ("redeem didn't
  work") or several paragraphs ("I tried to redeem my coupon at X but got
  a code that expired before...").
- **ticket:SCRUM-123**: fetch the ticket via Atlassian MCP. Use its
  description + latest comment as the raw report.
- **email:thread-id**: fetch thread via Gmail MCP. Use the first user
  message as the raw report.

Never proceed without the raw text. If fetch fails, stop and ask for a
paste.

### 2. Collect context the user already gave

Extract from the raw report:

- User-facing symptom ("button didn't work", "got an error", "money
  charged but no confirmation")
- Workflow affected (redemption, donation, signup, etc.)
- Actor type if mentioned (business owner, consumer, staff)
- Reproducer details (device, browser, timestamp, specific product/offer)
- Error messages if the user pasted them verbatim

If any of these are genuinely ambiguous and the fix would depend on it
(e.g., "didn't work" could mean crash or silent failure), flag as
`needs_user_clarification` and produce follow-up questions for the human
to ask the user BEFORE triage continues.

### 3. Run diagnostic inference

Dispatch the `architect` agent in diagnostic mode with the raw report.
Architect's job here is different from planning — it's reverse-engineering
where the bug lives. Architect should produce:

- **Most likely root cause** with a confidence level (high / medium / low)
- **Spec section** that governs the broken behavior (e.g.,
  `docs/specs/schema/05-offers-events.md §7 Redemptions`)
- **Candidate files** where the bug may exist, verified by grep for
  relevant function names / error strings / matching patterns
- **Existing tests** that should have caught this (and why they didn't —
  missing case? actor type not tested? edge condition?)
- **Severity** per the rubric in `.claude/agents/qa-tester.md`:
  - **critical** — data loss, payment wrong, security breach, data leak
  - **high** — user-blocking for a core workflow
  - **medium** — degraded UX, workaround exists
  - **low** — cosmetic or extreme edge case

If architect returns `confidence: low`, do not auto-assign a severity —
ask the human to triage by hand.

### 4. Check for duplicates

JQL search across open bugs for similar symptoms:

```
project = SCRUM AND issuetype = Bug AND status != Done
  AND (summary ~ "<key symptom words>" OR description ~ "<key symptom words>")
```

For each candidate duplicate, fetch and compare:

- Same workflow?
- Same error message or close variant?
- Same spec section?

If confidence > 70% it's a duplicate, STOP and report: "This looks like a
duplicate of SCRUM-XXX. Link as duplicate instead of creating new?"

The human decides whether to create anyway (if it's actually distinct) or
link to the existing ticket.

### 5. Draft the ticket

Using the template in `.claude/templates/jira-ticket-template.md` with
bug-specific additions:

```markdown
# <auto-generated summary in engineering language>

## Description

<One paragraph, engineering framing. Example:
"Redemption flow produces a code that expires before the user can reach
the counter. Reported by consumer on iOS; reproducer shows the 10-min
countdown starts before the proximity check completes server-side,
causing the timer to be effectively shorter than intended.">

## Reported by user

<Verbatim user report, preserved exactly. This is what the user said,
in their own words. Do not edit for style.>

## Severity

<critical | high | medium | low>

<One-sentence rationale.>

## Acceptance criteria

- [ ] <specific, testable outcome>
- [ ] Regression test added per Test cases below
- [ ] User who reported the bug notified of resolution

## Spec references

<Paths into docs/specs/ with anchors, from architect's diagnosis>

## Files touched

<From architect's candidate files list. Start with read-only for
investigation; promote to write: after root cause is confirmed.>

**write:** _(to be confirmed during investigation)_
  - <candidate file 1>
  - <test file that will hold the regression test>

**read-only:**
  - <any file that might need to be inspected>

## Test cases

**New (required — must fail before fix, pass after):**
  - <domain>.<feature>.regression.<scenario>_<actor>

**Existing tests that should have caught this:**
  - <existing test ID if any>
  <comment on why it didn't catch the bug — missing case? actor gap?>

## Reproducer

1. <step>
2. <step>
3. <expected vs actual>

## Environment

- App: <mobile iOS / mobile Android / web / admin>
- Tenant: <main tenant, or tenant key if multi-tenant>
- Time of report: <iso-timestamp>
- User's actor type: <consumer / business / nonprofit / staff>

## Investigation notes

<Space for the implementer to add findings during work.
Starts empty.>
```

### 6. Present for human review

Output the full draft plus:

- Severity rationale
- Confidence level from architect
- List of duplicate candidates (even if below threshold — humans sometimes
  see similarities the search missed)
- Proposed Jira priority: critical → Highest, high → High, medium →
  Medium, low → Low
- Proposed labels: always `bug`, always `reported-by-user`, plus the
  `phase-*` label matching the affected spec section, plus `regression`
  if an existing test failed to catch this

Ask: "Create this ticket? [y/n/edit]"

If `edit`: enter an interactive mode where the human can adjust severity,
tighten the summary, correct file paths, add/remove labels before
creating.

If `y`: proceed to creation (step 7).

If `n`: stop, don't create.

If `--apply` was passed: skip this step and go straight to creation with
default proposed values. Emit a warning that human review was skipped and
log to `.claude/audit.log`.

### 7. Create the ticket

Use Atlassian MCP. Cloud ID: `<FILL IN — from getAccessibleAtlassianResources>`.
Project key: `<FILL IN — usually SCRUM>`.

- Issue type: **Bug** (not Story)
- Summary: from the engineering-language summary line
- Description: the full drafted body from step 5
- Priority: proposed value
- Labels: proposed set

After creation, immediately:

- If user source was an existing ticket, link as "Relates to"
- If duplicate candidates were flagged but human said "create anyway",
  link as "Relates to" for each
- If the bug appears to block other in-progress tickets (architect
  identified transitive impact), link as "Blocks"

### 8. Close the loop with the reporter

If the source was a support ticket, email thread, or in-app report, add
a comment on that original artifact:

> Bug triaged into SCRUM-YYY. Status: <status>. We'll update you when the
> fix ships. Thank you for the report.

Template variants:

- **Support ticket**: post an internal comment linking to SCRUM-YYY; the
  support agent will relay to the user
- **Email**: draft a reply for the human to send; do NOT auto-send
- **In-app report**: add a comment on the corresponding
  `location_change_requests` or `tickets` row

### 9. Record

Append to `docs/bug-triage-log.md` one line:

```
YYYY-MM-DD <triager> <source> SCRUM-YYY <severity> <confidence> <summary>
```

This is a paper trail so triage quality can be audited — over time the
ratio of correct severity calls and confidence levels should improve.

## Error handling

- **Architect returns zero candidate files** — bug may not be in the
  codebase (third-party issue, user error, misunderstanding). Ask the
  human: "Architect couldn't locate this in the code. Possible
  interpretations: (a) external issue, (b) user misunderstanding, (c)
  missing diagnostic info. Which?" Don't auto-create.
- **Duplicate check returns >5 candidates** — symptom is too generic to
  search by; report the top 3 and ask human to confirm none match before
  creating.
- **Jira creation fails** — save draft to
  `.claude/drafts/bug-triage-<timestamp>.md` so the work isn't lost; the
  human can retry.
- **Severity disagreement** — if human selects a different severity from
  architect's proposal, log both in the audit record. Over time this
  trains architect's rubric.

## What this does NOT do

- Does not attempt to fix the bug. Creates the ticket, stops.
- Does not automatically assign the ticket. Assignment is a human call
  based on team capacity and expertise.
- Does not set sprint, fix-version, or estimate. Leave those to the
  team's planning process.
- Does not close the original support ticket — the support agent closes
  after the user confirms resolution.
- Does not prioritize the ticket in the backlog. Severity maps to priority
  but actual scheduling is `/batch-plan` and human judgment.

## Safety

- **Never auto-apply in production without `--apply`.** Human review is
  the default.
- **Never infer financial impact into severity without evidence.** A
  report mentioning "charged twice" doesn't guarantee it happened —
  architect must find corroborating evidence (duplicate `payments` rows,
  webhook log anomaly) before severity is escalated to critical.
- **Never expose user PII in the ticket summary.** User name, email,
  phone, payment info all stay in the "Reported by user" block and get
  redacted when the ticket is indexed for search. Summary stays generic:
  "Redemption timer shorter than 10 min" not "Redemption timer shorter
  than 10 min for user jane@example.com".
- **Never escalate severity above what architect recommends without a
  human's explicit override.** Under-reaction is the default posture;
  over-reaction dilutes the severity signal for the whole backlog.

## Interaction with the rest of the system

- Drafted tickets feed into `/batch-plan` like any other ticket once
  created
- Regression test requirement means `qa-tester` in the dispatcher's
  Phase 3 checks that the new test exists and fails before the fix
  lands — this is the anti-regression guarantee
- Severity maps to priority maps to `/batch-plan` wave assignment —
  critical bugs jump to the front of the queue
